# 프롬프트 이력

## 턴 1

### 입력

`init`

### 결과

- 빈 풀이 디렉터리를 초기화하고 작업 시작 상태를 만들었다.

## 턴 2

### 입력

`한글로 가자`

### 결과

- 이후 작업과 문서화를 모두 한국어 기준으로 진행하도록 방향을 전환했다.

## 턴 3

### 입력

`지금 프로젝트를 진행하려는데 먼저 problem.md파일을 읽고, 제출물 구조를 만들자. 그리고 scr엔 spring boot, gradle 프로젝트를 만들어서 문제를 해결하려해. 그리고 docs/문제분석.md 파일을 만들어서 문제 분석과 문제를 해결하는데 중요한 체크리스트를 만들어줘. 그리고 prompts/prompts.md파일엔 내가 입력한 프롬프트와 입력 결과물에 대한 요약을 작성해줘.`

### 결과

- `PROBLEM.md`를 읽고 과제의 핵심 요구사항과 구현 우선순위를 정리했다.
- 제출물 구조에 맞춰 루트에 `README.md`, `AGENTS.md`, `docs/`, `prompts/`를 구성했다.
- 문서 초안과 Spring 프로젝트 초기화 방향을 잡았다.

## 턴 4

### 입력

`지금 src/schoolSystem이란 이름으로 내가 프로젝트를 생성해둔게 있어. 거기서 코드작성 진행하자`

### 결과

- 실제 구현 기준을 사용자가 미리 만든 `src/schoolSystem` 프로젝트로 전환했다.
- `src/schoolSystem`을 Spring Boot + Gradle 기반으로 맞추고 `/health` 엔드포인트와 기본 테스트를 추가했다.
- `./gradlew test` 실행이 성공하는 상태까지 확인했다.

## 턴 5

### 입력

`좋아 그런데 그전에 prompts.md파일은 입력값에 대한 결과물이 하나씩 매칭되도록 입력1, 결과1, 입력2, 결과2 .. 같은 방식으로 가자. 그리고 문제분석과 체크리스트파일 분리하자. 체크리스트도 지금 나열식으로 되어있는데 주제별로 묶어서 세부 사항도 점검할 수 있게 하자. 예를들면 초기데이터 생성의 경우 1분내 생성, 학과 10개이상 생성등으로 묶어서. 그리고 지금 docs파일에 requirements.md파일이 빠졌어`

### 결과

- 프롬프트 이력을 입력/결과 쌍 구조로 재정리했다.
- `문제분석`과 `체크리스트` 문서를 분리했다.
- 체크리스트를 주제별 점검 항목 구조로 재작성했다.
- 빠져 있던 `docs/REQUIREMENTS.md`를 추가했다.

## 턴 6

### 입력

`예시를 입력 1, 결과 1 같이 말했는데 이건 오히려 보기 불편하잖아. 좀 더 보기 깔끔한 구조로 가자. 그리고 requirements.md파일에 먼저 작성하려했던 도메인 모델 먼저 보여줄래`

### 결과

- 프롬프트 이력을 턴 단위 구조로 재정리했다.
- `docs/REQUIREMENTS.md` 상단에 도메인 모델 초안을 먼저 배치했다.

## 턴 7

### 입력

`좋아 일단 db는 h2로 가자. enrolled를 파생값으로 둘지와 별도 상태로 둘지가 어떤거지? 충돌은 시간단위로 가자.`

### 결과

- 저장소를 H2 기반으로 가져가기로 결정했다.
- 시간 충돌 검증 단위를 교시가 아닌 실제 시간 단위로 정리했다.
- `enrolled`를 파생값으로 계산하는 방식과 `Course` 상태값으로 저장하는 방식의 차이를 비교했다.

## 턴 8

### 입력

`일단 동시성 전략을 어떻게 가져가려했는지 부터 보여줄래? 그걸 봐야 결정할 수 있을거같은데.`

### 결과

- `Student`와 `Course`를 함께 잠그는 비관적 락 기반 전략을 설명했다.
- 같은 학생의 동시 신청과 같은 강좌의 좌석 경쟁을 동시에 제어하려는 방향을 공유했다.

## 턴 9

### 입력

`course를 잠글거면 집계보단 course에 enrolled를 두는게 더 효율적이지않나? 아님 다른 문제가 있나?`

### 결과

- `Course.enrolledCount` 상태값으로 관리하면 신청 경로가 단순하고 빠르다는 점을 정리했다.
- 대신 신청 내역과 상태값을 함께 갱신해야 하므로 트랜잭션과 락이 더 중요해진다는 점을 설명했다.

## 턴 10

### 입력

`그럼 취소시에도 락을 잡을필요가 있나? 그리고 지금 조건이 학생이 최소 만명인데  만개의 요청이 들어왔을때 서버가 버틸수있나?`

### 결과

- 취소도 정합성을 위해 같은 자원 보호 범위 안에 두는 것이 안전하다고 설명했다.
- 다만 이 전략은 고성능보다 정확성 우선 전략이며, H2 단일 서버에서 초고성능을 목표로 하지 않는다는 점을 정리했다.

## 턴 11

### 입력

`좋아 성능 최적화는 일단 제쳐두고, 정합성 보장만 가자. 그리고 취소의 경우 enrolled때문에 coursed에도 락을 잡게 되는건데 이게 이득으로 보여 아님 손해로보여?`

### 결과

- 성능보다 정합성을 우선한다는 기준을 확정했다.
- `enrolledCount`를 유지하면 취소에서도 `Course` 락을 잡는 것이 구조적으로 더 일관된다는 점을 설명했다.

## 턴 12

### 입력

`아니 enrollcount때문에 잡게 되는거니. enrollecount를 빼면 신청때는 db 조회후 정원 초과 판단하게 되는데 대신 취소땐 락을 잡을 필요가 없어지니. 평균적으로 한 학생당 취소는 2~3건만 진행된다 가정한다면`

### 결과

- 파생값 방식이면 취소 경로가 단순해지는 장점이 있다는 점을 검토했다.
- 신청의 핵심성과 취소 빈도를 기준으로 두 방식의 trade-off를 다시 비교했다.

## 턴 13

### 입력

`신청때가 핵심인데 enrolledcount를 두는게 db조회보다 더 효율적이지 않을까 했어서`

### 결과

- 신청 경로만 보면 `enrolledCount` 상태값이 집계 조회보다 더 효율적이라는 점을 명확히 확인했다.
- 신청 효율과 취소 단순성 중 어디에 복잡도를 둘지 판단 기준을 정리했다.

## 턴 14

### 입력

`음.. 아님 enrollcount를 메모리에 저장하는건 별로려나`

### 결과

- 메모리 상태값은 DB와 정합성이 어긋날 위험이 커서 현재 목표와 맞지 않는다고 정리했다.
- `enrolledCount`를 둘 거면 DB 컬럼으로 두고 트랜잭션 안에서만 갱신하는 방향으로 좁혔다.

## 턴 15

### 입력

`아니 그럼 enrollcount 추가하는 방식으로 해서 코드작성 가보자. 그리고 지금 prompts.md파일 업데이트가 왜 안되었어? 코드 작성을 하지 않더라도 사고 과정도 기록하려해서`

### 결과

- 최근 설계 토론이 빠진 것은 기록 누락이었다는 점을 확인하고, 이후 설계 판단도 모두 기록 대상으로 포함하기로 했다.
- `Course.enrolledCount`를 포함한 H2 + JPA 모델과 락 전략을 문서에 반영하고 코드 작업을 시작하기로 했다.


## 턴 16

### 입력

`음 디펜던시에 lombok이랑 swagger, junit, assertj, validation 추가하고 그에 맞게 코드도 수정가자.`

### 결과

- Gradle 의존성에 Lombok, Swagger, Validation, JUnit, AssertJ를 명시적으로 추가했다.
- 엔티티 코드를 Lombok 기반으로 정리하고, 헬스체크 API에 Swagger 어노테이션과 응답 DTO를 추가했다.
- 테스트 코드를 AssertJ 기반 `TestRestTemplate` 방식으로 변경했다.


## 턴 17

### 입력

`지금 이 에러가 뜨며 앱이 실행 안되는데`

### 결과

- 직접 `./gradlew bootRun`으로 실행 로그를 확인했다.
- 실제 실패 원인은 `8080` 포트 충돌이었다.
- 추가로 `DayOfWeek` enum이 H2 PostgreSQL 모드에서 `tinyint`로 생성되며 경고가 발생해 `EnumType.STRING`으로 수정했다.


## 턴 18

### 입력

`아니 실행 확인했어. 그러면 이제 나머지도 작성가보자. 그리고 작성할때 지금 에러가 이렇게 되어있으면 어디서 에러났는지 확인이 귀찮아져. 에러코드를 따로 관리해서 도메인별로 묶고, 어디서 에러가 나는지 쉽게 확인할 수 있게 하자. 에러코드엔 메시지와 httpcode가 포함되어 있어야해. 그리고 dto는 따로 폴더에 만들어서 쉽게 볼 수 있게 하고, api패키지도 domain 패키지처럼 도메인별로 묶어서 쉽게 관리할 수 있게 하자. 그리고 공통코드는 global이나 common 패키지에 관리하자`

### 결과

- `global` 패키지에 공통 예외 처리와 에러 응답 구조를 추가했다.
- 도메인별 에러코드를 분리하고, `api/<domain>/dto` 구조로 패키지를 재정리했다.
- 학생/교수/강좌 조회, 시간표 조회, 수강신청/취소를 위한 기본 서비스와 컨트롤러 뼈대를 추가했다.
- 초기 데이터 생성기도 함께 추가해 실행 시 기본 데이터가 준비되도록 했다.


## 턴 19

### 입력

`구조를 정리한 뒤 나머지도 작성하자`

### 결과

- 수강신청 성공과 중복신청 실패를 검증하는 통합 테스트를 추가했다.
- API 문서에 현재 구현된 엔드포인트와 공통 에러 응답 구조를 반영했다.

## 턴 20

### 입력

`일단 서비스는 api 패키지에 두자`

### 결과

- 서비스 계층을 `domain/.../service`에서 `api/<domain>/service`로 이동했다.
- 컨트롤러 import와 참조 경로를 새 패키지 기준으로 정리했다.


## 턴 21

### 입력

`course 예외도 errorcode로 관리하고, 문제분석 결정 사항을 정리하자`

### 결과

- `Course` 내부 예외를 `CourseErrorCode` 기반 `BusinessException`으로 통일했다.
- 인증/인가 제외, 학생 ID 직접 입력, 중복신청/취소대상없음 정책, 시간 충돌 판정 기준을 문서에 반영했다.
- 시간 충돌은 DB가 후보를 줄이고 최종 판정은 도메인 로직이 맡는 방향으로 정리했다.


## 턴 22

### 입력

`좋아 바로 가보자`

### 결과

- 시간표 충돌 검증에서 DB는 같은 요일 후보만 조회하고, 최종 겹침 판정은 도메인 로직이 수행하도록 리포지토리와 서비스를 수정했다.


## 턴 23

### 입력

`좋아 테스트도 바로 추가가자`

### 결과

- 취소 대상 없음, 시간 충돌, 학점 초과, 정원 초과 케이스를 포함하는 통합 테스트를 추가했다.
- 취소 대상이 없을 때의 메시지를 방금 정한 정책에 맞게 `취소할 강의가 없습니다.`로 수정했다.


## 턴 24

### 입력

`데이터 중복이 너무 많고 조회 성능을 위해 인덱싱도 필요하다`

### 결과

- 학생, 교수, 강좌, 수강신청 엔티티에 조회 키 중심 인덱스를 추가했다.
- 이름과 강좌명은 중복을 허용하되 비유니크 인덱스로 검색 성능을 보완하도록 정리했다.
- 초기 데이터 생성 토큰 풀을 크게 늘려 이름과 강좌명의 다양성을 높였다.


## 턴 25

### 입력

`동시성 테스트와 테스트 구조 분리를 하자`

### 결과

- 테스트를 `unit`, `integration`, `concurrency` 패키지로 분리했다.
- 단위 테스트는 시간표 충돌 로직, 통합 테스트는 API 규칙 검증, 동시성 테스트는 멀티스레드 좌석 경쟁 검증으로 역할을 나눴다.
- 잔여 좌석 1개 상황에서 여러 스레드가 동시에 신청해도 1건만 성공하는 동시성 테스트를 추가했다.


## 턴 26

### 입력

`문제분석에는 결정이 필요한 사항만 두고, 현재 결정은 REQUIREMENTS로 옮기자`

### 결과

- `문제분석.md`에서는 현재 결정 내용을 제거하고, 판단이 필요했던 항목만 남기도록 정리했다.
- 인증/식별/중복신청/취소정책/시간충돌/동시성/에러처리/인덱스 전략 같은 현재 결정은 `REQUIREMENTS.md`로 모아 재구성했다.

## 턴 27

### 입력

`지금까지 구현한거 잘 되었는지 테스트를 통해서 확인해보려하는데 우선 지금 테스트에선 어떤걸 테스트하는지 docs/test.md파일에 정리해줄래?`

### 결과

- 현재 테스트를 단위 테스트, 통합 테스트, 동시성 테스트로 구분해 어떤 규칙을 검증하는지 문서로 정리했다.
- 각 테스트 파일의 목적, 실행 방법, 현재 커버하는 범위와 아직 비어 있는 시나리오를 `docs/test.md`에 기록했다.

## 턴 28

### 입력

`좋아 그럼 이제 체크리스트에 있는 요소들을 테스트에 모두 추가헤보자.`

### 결과

- 초기 데이터, 조회 API, 취소 후 좌석 복구, 같은 학생 동시 신청 제어까지 체크리스트 기준으로 테스트를 확장했다.
- `DataInitializationIntegrationTest`, `QueryApiIntegrationTest`를 추가하고 기존 수강신청/동시성 테스트를 보강했다.
- 전체 `./gradlew test` 실행이 성공했고, 현재 자동화 범위와 수동 확인이 필요한 항목을 `docs/test.md`에 다시 정리했다.

## 턴 29

### 입력

`지금 코드에서도 직접 확인해보려하는데 displayname 추가해주고, 테스트에 주석달아서 체크리스트의 어떤걸 체크했는지도 적어줄래?`

### 결과

- 모든 테스트 메서드에 `@DisplayName`을 추가해 테스트 리포트와 코드에서 의도를 바로 읽을 수 있게 했다.
- 각 테스트 메서드 앞에 체크리스트의 어떤 항목을 검증하는지 주석으로 남겼다.
- 변경 후 `./gradlew test`를 다시 실행해 전체 테스트가 여전히 통과하는 것을 확인했다.

## 턴 30

### 입력

`지금 데이터 생성에서 1분이내 초기 데이터 생성이 되는지와 끝난 후에 health가 200을 반환하는지는 테스트가 안되었는데. 데이터가 준비되기 전까진 health나 다른 api 호출이 안되어야해`

### 결과

- 초기 데이터 준비 상태를 관리하는 `ApplicationReadinessState`와 요청 차단용 `ReadinessGuardFilter`를 추가했다.
- 초기 데이터 준비가 끝나기 전에는 `health`를 포함한 모든 API가 `COMMON_503`으로 차단되고, 준비 완료 후에만 호출 가능하도록 바꿨다.
- 초기 데이터 생성 1분 이내 완료 테스트와 준비 전/후 API 차단 테스트를 추가했고 `./gradlew test` 전체 통과를 확인했다.

## 턴 31

### 입력

`지금 동시성 테스트보는데 왜 순차적으로 실행하게 되어있지? 각각 다른 스레드에서 실행했을때를 비교해야하지않나?`

### 결과

- 동시성 테스트에서 순차 실행처럼 보이던 부분은 사전 상태를 만드는 준비 단계라는 점을 분리해서 드러냈다.
- `EnrollmentConcurrencyTest`를 준비 단계와 실제 멀티스레드 동시 실행 단계로 나눠 읽히도록 리팩터링하고 주석을 보강했다.
- 리팩터링 후 해당 동시성 테스트를 다시 실행해 통과를 확인했다.

## 턴 32

`좋아 그럼 이제 마무리 정리 가자. prompts.md의 내용을 바탕으로 requirements.md파일을 보완해주고, api.md문서도 업데이트하자`
