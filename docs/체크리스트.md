# 구현 체크리스트

## 1. 프로젝트 실행 가능 상태

- [ ] `src/schoolSystem`에서 `./gradlew bootRun`이 실행된다.
- [ ] 애플리케이션 기동 직후 오류 없이 서버가 유지된다.
- [ ] `GET /health`가 200을 반환한다.
- [ ] `GET /health` 응답 형식이 문서와 일치한다.

## 2. 초기 데이터 생성

- [ ] 애플리케이션 시작 후 1분 이내에 초기 데이터 생성이 완료된다.
- [ ] 학과가 10개 이상 생성된다.
- [ ] 교수 데이터가 100명 이상 생성된다.
- [ ] 학생 데이터가 10,000명 이상 생성된다.
- [ ] 강좌 데이터가 500개 이상 생성된다.
- [ ] 생성 데이터가 의미 있는 이름과 패턴을 가진다.
- [ ] 생성 데이터가 비즈니스 규칙을 위반하지 않는다.
- [ ] 초기 데이터 준비가 끝난 시점에 조회 API 호출이 가능하다.

## 3. 조회 API

- [ ] 학생 목록 조회 API가 동작한다.
- [ ] 교수 목록 조회 API가 동작한다.
- [ ] 강좌 목록 조회 API가 동작한다.
- [ ] 학과별 강좌 조회가 가능하다.
- [ ] 강좌 목록 응답에 id가 포함된다.
- [ ] 강좌 목록 응답에 name이 포함된다.
- [ ] 강좌 목록 응답에 credits가 포함된다.
- [ ] 강좌 목록 응답에 capacity가 포함된다.
- [ ] 강좌 목록 응답에 enrolled가 포함된다.
- [ ] 강좌 목록 응답에 schedule이 포함된다.
- [ ] 내 시간표 조회 API가 동작한다.

## 4. 수강신청 규칙

- [ ] 수강신청 API가 동작한다.
- [ ] 수강취소 API가 동작한다.
- [ ] 동일 강좌 중복 신청이 차단된다.
- [ ] 18학점 초과 신청이 차단된다.
- [ ] 시간표 충돌 신청이 차단된다.
- [ ] 정원이 가득 찬 강좌 신청이 차단된다.
- [ ] 취소 후에는 다시 잔여 좌석이 반영된다.

## 5. 동시성 제어

- [ ] 잔여 좌석 1개인 강좌에 동시 요청이 들어와도 1명만 성공한다.
- [ ] 동시에 여러 요청이 와도 enrolled 값이 capacity를 초과하지 않는다.
- [ ] 동일 학생이 동시에 여러 강좌를 신청할 때 학점 제한이 깨지지 않는다.
- [ ] 동일 학생이 동시에 여러 강좌를 신청할 때 시간 충돌 제한이 깨지지 않는다.
- [ ] 동시성 제어 방식과 락 범위를 문서로 설명했다.

## 6. 문서화

- [ ] `docs/REQUIREMENTS.md`에 요구사항 해석과 설계 결정을 정리했다.
- [ ] `docs/API.md`에 엔드포인트별 요청/응답을 정리했다.
- [ ] `docs/문제분석.md`에 문제 해석과 위험 요소를 정리했다.
- [ ] `prompts/prompts.md`에 입력/결과 쌍 형태로 프롬프트 이력을 정리했다.
