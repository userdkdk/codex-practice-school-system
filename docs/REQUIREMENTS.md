# REQUIREMENTS

## 1. 과제 목표

이 과제의 핵심은 대학교 수강신청 백엔드를 구현하면서, 동시 신청 상황에서도 정원 초과를 절대 허용하지 않는 것이다. 단순 조회 기능보다 수강신청 처리의 정합성, 동시성 제어, 초기 데이터 준비 상태 관리가 더 중요하다.

## 2. 도메인 모델

### Department

학과 정보다.

- `id`: 학과 ID
- `name`: 학과명

### Professor

교수 정보다.

- `id`: 교수 ID
- `name`: 교수명
- `departmentId`: 소속 학과 ID

### Student

학생 정보다.

- `id`: 학생 ID
- `studentNumber`: 학번
- `name`: 학생명
- `departmentId`: 소속 학과 ID
- `grade`: 학년

### CourseScheduleSlot

강의 시간표의 최소 충돌 판정 단위다.

- `dayOfWeek`: 요일
- `startTime`: 시작 시간
- `endTime`: 종료 시간

### Course

수강 가능한 강좌 정보다.

- `id`: 강좌 ID
- `code`: 강좌 코드
- `name`: 강좌명
- `departmentId`: 개설 학과 ID
- `professorId`: 담당 교수 ID
- `credits`: 학점
- `capacity`: 정원
- `enrolledCount`: 현재 신청 인원 상태값
- `scheduleSlots`: 시간표 슬롯 목록

### Enrollment

학생의 수강신청 내역이다.

- `id`: 신청 ID
- `studentId`: 학생 ID
- `courseId`: 강좌 ID
- `createdAt`: 신청 시각

## 3. 도메인 규칙

- 한 학생은 최대 18학점까지만 신청할 수 있다.
- 동일 학생은 동일 강좌를 중복 신청할 수 없다.
- 동일 학생은 시간이 겹치는 두 강좌를 동시에 신청할 수 없다.
- 강좌의 신청 인원은 정원을 초과할 수 없다.
- 수강취소가 되면 신청 내역이 제거되고 해당 강좌의 `enrolledCount`가 함께 감소해야 한다.

## 4. 기능 요구사항

- 서버는 `GET /health`를 제공해야 한다.
- 학생 목록 조회 기능이 있어야 한다.
- 교수 목록 조회 기능이 있어야 한다.
- 강좌 목록 조회 기능이 있어야 한다.
- 학과별 강좌 조회 기능이 있어야 한다.
- 수강신청 기능이 있어야 한다.
- 수강취소 기능이 있어야 한다.
- 학생별 이번 학기 시간표 조회 기능이 있어야 한다.

## 5. 비기능 요구사항

- 애플리케이션 시작 후 1분 이내에 초기 데이터가 준비되어야 한다.
- 초기 데이터는 런타임에 동적으로 생성되어야 한다.
- 초기 데이터 규모는 학과 10개 이상, 강좌 500개 이상, 학생 10,000명 이상, 교수 100명 이상이어야 한다.
- API 문서는 평가자가 바로 테스트를 작성할 수 있을 정도로 구체적이어야 한다.
- 동시 요청 상황에서도 정원 초과가 발생하지 않아야 한다.
- 초기 데이터 준비가 완료되기 전에는 `health`를 포함한 어떤 API도 호출되면 안 된다.

## 6. 현재 설계 결정

### 기술 스택

- Java 21
- Spring Boot
- Gradle
- H2
- Spring Data JPA
- Spring Validation
- Springdoc OpenAPI
- Lombok

### 인증/인가 및 식별 방식

- 인증/인가는 우선 제외한다.
- 수강신청 및 수강취소 요청은 학생 ID를 직접 받는다.
- 학생 식별 방식은 ID 기준으로 통일한다.

### 수강신청/취소 정책

- 동일 강좌 중복 신청 시 이미 신청한 강좌라는 에러를 반환한다.
- 취소 대상이 없으면 취소할 강의가 없다는 의미의 에러를 반환한다.
- 강좌의 현재 신청 인원은 `Course.enrolledCount` 상태값으로 관리한다.
- 수강취소 후에는 `Enrollment` 삭제와 `Course.enrolledCount` 감소가 함께 반영되어야 한다.

### 시간표 충돌 판정

- 시간 충돌 검증은 교시 기준이 아니라 실제 시간 단위로 처리한다.
- DB는 같은 요일의 후보를 줄이는 역할까지만 맡는다.
- 최종 시간 충돌 판정은 도메인 로직에서 처리한다.
- 충돌 기준은 같은 요일에서 시간이 실제로 겹치는지 여부다.

### 동시성 제어

- 수강신청은 `Student`와 `Course`를 같은 순서로 비관적 락 조회한 뒤 처리한다.
- 같은 학생의 동시 신청에서도 학점 제한과 시간 충돌 검증이 깨지지 않도록 `Student` 락을 사용한다.
- 같은 강좌에 대한 동시 신청에서도 정원 초과가 발생하지 않도록 `Course` 락과 `enrolledCount` 갱신을 같은 트랜잭션에서 처리한다.
- 수강취소도 `Student`와 `Course`를 같은 순서로 잠그고 `Enrollment` 삭제와 `enrolledCount` 감소를 함께 처리한다.
- 성능 최적화보다 정합성 보장을 우선한다.

### 초기 데이터 준비 상태 정책

- 애플리케이션은 초기 데이터 생성 시작 시 준비 중 상태로 들어간다.
- 초기 데이터 생성이 끝나기 전까지 `health`를 포함한 모든 API는 `503 Service Unavailable`을 반환한다.
- 초기 데이터 생성이 완료된 뒤에만 `health`가 `200 OK`를 반환하고 다른 API도 호출 가능해진다.
- 초기 데이터 생성 시간은 애플리케이션 내부에서 기록하고 테스트로 1분 이내 완료 여부를 검증한다.

### 데이터 생성 및 인덱스 전략

- 초기 데이터는 현실적인 이름과 강의명 패턴을 가지도록 생성한다.
- 학생 이름, 교수 이름, 강좌명은 중복을 허용한다.
- 중복 가능 컬럼에도 비유니크 인덱스를 사용할 수 있으며, 조회 성능 보완을 위해 적용한다.
- `studentNumber`, `course.code`, `student_id`, `course_id` 같은 주요 조회 키에는 인덱스를 둔다.

### 에러 처리 기준

- 에러는 공통 응답 형식을 사용한다.
- 도메인별 에러코드는 `message`, `httpStatus`, `code`를 함께 가진다.
- 도메인 내부에서 발생하는 예외도 가능한 한 도메인 에러코드로 통일한다.
- 초기 데이터 준비 중 요청은 `COMMON_503` 에러코드로 응답한다.

## 7. 패키지 구조 기준

- API 계층은 `api/<domain>` 기준으로 관리한다.
- 요청/응답 DTO는 `api/<domain>/dto`에 둔다.
- 서비스는 `api/<domain>/service`에 둔다.
- 도메인 엔티티와 리포지토리는 `domain/<domain>`에 둔다.
- 공통 코드는 `global` 패키지에 둔다.
- 준비 상태 제어는 `global/readiness`에서 관리한다.

## 8. 테스트 기준

현재 자동화 테스트는 아래 범위를 커버한다.

- 헬스체크 응답
- 초기 데이터 최소 개수
- 초기 데이터 생성 시간 1분 이내 여부
- 초기 데이터 준비 전 API 차단 여부
- 학생/교수/강좌 조회 API
- 학과별 강좌 조회
- 수강신청/수강취소 및 시간표 조회
- 중복신청, 학점 초과, 시간 충돌, 정원 초과 차단
- 취소 후 좌석 복구
- 같은 강좌 경쟁 상황의 동시성 제어
- 같은 학생 동시 신청 상황의 학점/시간 충돌 제어
