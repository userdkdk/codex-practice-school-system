# 문제 분석

## 1. 과제의 본질

이 과제의 핵심은 단순한 CRUD 서버를 만드는 것이 아니라, 수강신청이라는 경쟁 상황에서 정원 초과를 절대 허용하지 않는 백엔드 시스템을 구현하는 것이다. 특히 잔여 좌석이 1개인 강좌에 다수의 요청이 동시에 들어와도 정확히 1명만 성공해야 하므로, 조회 기능보다 수강신청 처리의 원자성과 일관성이 더 중요하다.

## 2. 명시된 요구사항 정리

- `GET /health`는 반드시 200 OK를 반환해야 한다.
- 학생 목록 조회 API가 필요하다.
- 교수 목록 조회 API가 필요하다.
- 강좌 목록 조회 API가 필요하다.
- 수강신청 API가 필요하다.
- 수강취소 API가 필요하다.
- 내 시간표 조회 API가 필요하다.
- 학생당 최대 신청 가능 학점은 18학점이다.
- 동일 시간대 강좌는 동시에 신청할 수 없다.
- 서버 시작 시점에 초기 데이터가 준비되어 있어야 한다.
- 초기 데이터 준비는 1분 이내에 끝나야 한다.
- 학과 10개 이상, 강좌 500개 이상, 학생 10,000명 이상, 교수 100명 이상이 필요하다.
- 데이터는 정적 파일이 아니라 실행 시점에 동적으로 생성되어야 한다.
- API 문서는 `docs/` 아래에 작성되어야 한다.

## 3. 아직 결정이 필요한 사항

아래 항목들은 현재 구현과 문서를 더 다듬기 위해 추가 판단이 필요한 지점들이다. 이미 확정된 정책은 `docs/REQUIREMENTS.md`에서 관리한다.

- 조회 API의 검색 조건을 어느 범위까지 열지
- 조회 API의 페이지 크기 기본값과 최대값을 어떻게 둘지
- 초기 데이터 생성 시 학과별 학생/교수/강좌 분포를 더 현실적으로 조정할지
- 동시성 테스트 범위를 같은 강좌 경쟁 외에 같은 학생 동시 신청, 신청/취소 동시 실행까지 넓힐지
- Swagger 문서에 예시 요청/응답과 에러 사례를 어느 수준까지 넣을지

## 4. 구현 우선순위 해석

1. 서버가 정상 기동하고 `GET /health`가 성공해야 한다.
2. 평가자가 유효한 ID를 찾을 수 있도록 조회 API가 준비되어야 한다.
3. 수강신청과 취소가 비즈니스 규칙을 지키며 동작해야 한다.
4. 동시 요청에서도 정원 초과가 발생하지 않아야 한다.
5. 요구사항 해석과 설계 판단을 문서로 남겨야 한다.

## 5. 위험 요소

- 초기 데이터 생성이 느리면 평가 시작점부터 실패할 수 있다.
- 시간표 충돌 데이터가 비현실적이면 검증 가치가 떨어진다.
- 강좌 단위만 잠그면 동일 학생의 동시 신청에서 학점/시간 충돌 검증이 깨질 수 있다.
- API 문서와 에러코드가 구체적이지 않으면 구현이 맞아도 평가자가 테스트하기 어렵다.
- 데이터 토큰 풀이 너무 좁으면 이름과 강좌명이 과하게 중복되어 현실성이 떨어지고 검색 품질도 낮아질 수 있다.
